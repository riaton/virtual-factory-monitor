# virtual-factory-monitor: 目的とスコープ

**版**: 0.3 / **更新**: 2026-07-07 / **状態**: Phase 0 — 主要設計判断確定(ADR-001〜003、docs/adr.md 参照)

---

## 1. 目的(なぜ作るか)

工場を持たない個人が IT/OT 融合を学ぶための「疑似工場」予知保全検証環境。

このリポジトリが証明するもの:

1. **縦貫の完遂力** — センサーからAIまで一気通貫で動くものを作り切る力(物理 × クラウド × AI の掛け算)
2. **制約下の判断** — 60時間という予算の中で、何を実装し何を文書に留めるかの線引きとその言語化
3. **現場への認識** — データの汚さ(欠損・重複・時刻ズレ)と疑似環境の限界への自覚

価値の基準は網羅性ではなく縦貫。読者(想定滞在時間90秒)がREADME 1枚で上記を受け取れること。

## 2. スコープ(作るもの)

```
疑似センサー(Docker) → AWS IoT Core ─→ ブリッジ(購読) → InfluxDB(ローカル) ─→ Grafana(ローカル)
                            └→ SiteWise                              └→ Python CLI → Bedrock
```

| 要素 | 内容 |
|---|---|
| 疑似センサー | Dockerコンテナ × 3系統(温度・振動・稼働状態)。周期・ノイズ・故障モードを設定ファイルで注入可能 |
| 取り込み | AWS IoT Core(MQTT、X.509証明書認証) |
| ブリッジ | IoT Core を MQTT 購読し InfluxDB へ書き込む常駐サービス(IoT Thing として独自証明書を持つ)【ADR-001】 |
| 蓄積 | InfluxDB(ローカル、時系列)+ SiteWise(クラウド、アセットモデル)【ADR-001】 |
| 重複吸収 | InfluxDB の書き込み意味論(同一 measurement+タグ+時刻は同一ポイントへ上書き)【ADR-001】 |
| 可視化 | Grafana — ローカルの docker compose でセルフホスト【ADR-002】 |
| AI層 | Python CLI → InfluxDBクエリ → Bedrock で異常を日本語説明【ADR-003】 |
| IaC | Terraformで全構成をコード化 |
| AI駆動開発 | .claude/(CLAUDE.md・カスタムskill・agents設定)を同梱 |

### 一点豪華主義(唯一深く作り込む非機能)

**ネットワーク断・センサー欠損時の挙動**: デバイス側バッファリング → IoT Core 再送 → InfluxDB の書き込み意味論で重複を吸収。
専用の排除コンポーネントや状態ストアを置かず、重複耐性を**データモデル設計に織り込む**(タグとタイムスタンプの組がそのまま吸収のキーになる。データモデル図に判断コメントとして残す)。
異常系シーケンス図(Mermaid)1本をREADMEに掲載し、シミュレータの故障モード注入(重複送信・時刻ズレ)で「重複を投げても二重計上されない」ことを実演可能にする(詳細はADR-001)。

### 差別化の仕込み(3点)

1. Terraformによる全構成IaC化(再現性: `terraform apply` + `docker compose up` で誰でも動く)
2. コスト明細をREADMEに実測値で公開
3. 「実際の工場と違う点」を正直に書くセクション

## 3. 作らないものリスト(明示的スコープ外)

| 作らないもの | 理由 | 代替 |
|---|---|---|
| 本番級の非機能全般(マルチAZ、冗長化、監視体制 等) | 実装すると200時間コース。個人検証での本番級非機能は証明にならない | docs/「本番化する場合の非機能設計」1枚で認識を証明 |
| 実機ハードウェア(Raspberry Pi) | 調達・セットアップの時間リスクが60時間予算に未計上。読者の再現性も低下 | 初期スコープ外。マイルストーン②(42h)通過後に「後付けの実機デモ」として追加を判断 |
| 詳細設計書 | コードを正とする(乖離防止) | 設計判断の記録(ADR風)に判断理由のみ残す |
| 正常系シーケンス図 | 構成図で自明 | 異常系1本のみ作成 |
| 画面設計書・テスト仕様書 | 読者(90秒)の理解に不要 | — |
| 4系統目以降のセンサー | 縦貫の証明に3系統で十分 | — |
| ポートフォリオサイト等への寄り道 | 本計画の対象外 | — |

判定基準: **「この文書・機能がないと読者が誤解するか」**。Noなら作らない。

## 6. 前提・制約

- **予算**: 60時間(週2〜2.5時間、2026年7月末着手 → 2026年12月末リリース)
- **配分**: 実装6割 / 設計・文書3割 / リリース1割

- **副産物**: 各Phaseのハマりを記録し、Zenn記事3〜4本分の素材とする(二毛作)
- **AWSコスト**: 実測公開が前提のため、稼働期間中の課金を許容(Phase 4のコスト実測は数日稼働が必要、Phase 2以降と並走)
