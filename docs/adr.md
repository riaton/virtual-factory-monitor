## ADR-001: 時系列ストアはローカル InfluxDB + 自作ブリッジとする(重複吸収は書き込み意味論で)

**日付**: 2026-07-07 / **状態**: 承認

### 文脈
センサーデータの時系列蓄積層をどこに・何で持つか。本リポジトリの一点豪華主義(断・再送・重複吸収)の成立機構と、差別化の柱であるコスト実測公開の両方がこの選定に依存する。

### 選択肢
- (a) AWS のマネージド時系列ストア
- (b) SiteWise を蓄積・クエリ兼用にする(専用時系列ストアを置かない)
- (c) ローカル(docker compose)に InfluxDB を置き、IoT Core を MQTT 購読する自作ブリッジで書き込む

### 決定
**(c) を採用。** データは センサー → IoT Core(クラウド)→ ブリッジ(購読)→ InfluxDB(ローカル)と、クラウドを経由して手元に戻る往復経路をとる。

### 根拠
1. **重複吸収の成立**: InfluxDB は「同一 measurement + 同一タグセット + 同一タイムスタンプ」の書き込みを同一ポイントへの上書きとして扱う。デバイス発生時刻をポイントの時刻に採ることで、バッファからの再送 = 同一ポイント = 二重計上なし、が追加コンポーネントなしで成立する(設計本体は docs/data-model.md)。
2. **コスト構造**: 時系列ストアと可視化がローカルで完結するため、AWS 側の課金は IoT Core・SiteWise・Bedrock の従量分のみになる。コスト実測公開において「メッセージ量に比例する部分」だけが残り、読者に伝えたいコスト感が固定費に埋もれない。
3. **実工場との構図一致**: 製造現場では時系列データの一次蓄積をオンプレのデータヒストリアンが担い、クラウドは取り込み・モデル・分析を受け持つ分業が現実に多い。「クラウド経由でローカルのヒストリアンに落とす」本構成はその縮図であり、疑似工場の物語を強める。
4. **クエリの自由度**: 欠損検出(seq のギャップ分析)や遅延分析(受信時刻と発生時刻の差)など、一点豪華主義の観測に必要なアドホッククエリが制約なく書ける。

### 採らなかった案と理由
- **(a) AWS マネージド時系列ストア**: Amazon Timestream for LiveAnalytics は 2025 年 6 月 20 日をもって新規顧客への提供を終了しており、新規構築の選択肢に入らない。後継として案内される Timestream for InfluxDB はインスタンス常時稼働の固定費(月数十ドル規模)が発生し、根拠 2 のコスト構造と両立しない。
- **(b) SiteWise 兼用**: 構成要素は最少になるが、時系列のアドホッククエリ(ギャップ分析・遅延分析)が API の形に制約され、根拠 4 を満たせない。SiteWise はアセットモデル(設備の階層と語彙)の責務に限定して併用する。

### 受け入れたコスト
- ブリッジという自作常駐部品が 1 つ増える(IoT Thing として独自の X.509 証明書を持つ。デバイス 3 + 購読側 1 の証明書管理はむしろ実務の縮図として展示価値に転化する)。
- 「時系列蓄積までクラウド」という看板は掲げない。README ではローカル/クラウドの責務分担を構成図の境界線で明示する。

## ADR-002: 可視化はローカル Grafana(docker compose)とする

**日付**: 2026-07-07 / **状態**: 承認

### 文脈
可視化層の稼働場所。Amazon Managed Grafana(AMG)とローカルセルフホストが選択肢。

### 選択肢
- (a) Amazon Managed Grafana
- (b) ローカルの docker compose 内でセルフホスト

### 決定
**(b) を採用。** データソース定義とダッシュボード JSON は provisioning でリポジトリに同梱する。

### 根拠
1. **データ局所性**: 時系列ストア(InfluxDB)がローカルにある(ADR-001)以上、同一 compose 内での参照が最短・最simple。AMG からローカルの InfluxDB への到達はそもそも成立しない。
2. **コスト実測の見栄え**: AMG はユーザー単位の月額課金であり、固定費がコスト明細を支配して IoT パイプライン本体のコスト感が読者に伝わらなくなる。
3. **再現性**: `terraform apply` + `docker compose up` で可視化まで立ち上がり、ダッシュボード構築の手作業がゼロになる。

### 採らなかった案と理由
- **(a) AMG**: 本番でチームに提供するなら認証(SSO)・可用性・パッチ運用の観点で AMG が正しい。この要件差は「個人検証環境と本番の違い」の実例として docs/ の非機能設計文書に記載する。

## ADR-003: Bedrock チャットは Python CLI 直結とする(CLI → InfluxDB クエリ → Bedrock)

**日付**: 2026-07-07 / **状態**: 承認

### 文脈
AI 層(設備の異常を日本語で説明するチャット)のインターフェースとデータ経路。

### 選択肢
- (a) Web UI(Streamlit 等)を立てる
- (b) Grafana のパネル / プラグインに組み込む
- (c) Python CLI が InfluxDB に直接クエリし、結果を Bedrock に渡して日本語説明を生成する

### 決定
**(c) を採用。** CLI はコンテナ化し、compose の profiles で常駐サービス群から外して `docker compose run chat` で都度起動する。

### 根拠
1. **証明対象の純度**: 縦貫で証明したいのは「センサーデータ → AI による現場語の説明」というデータ経路であり、UI ではない。スコープの判定基準「この機能がないと読者が誤解するか」に対し、Web UI は No。
2. **工数**: UI 実装・ホスティングの工数(見積もりで 5h 超)を Phase 3 の 10h 枠から排除し、プロンプト設計の試行(4h)に予算を残す。
3. **デモ性**: CLI はターミナル録画で README のデモ GIF にしやすく、90 秒読者への訴求手段としてむしろ有利。

### 採らなかった案と理由
- **(a) Web UI**: 見栄えは上がるが証明対象に寄与しない。README に「Streamlit 等での UI 化は容易」と一行残し、拡張可能性の示唆に留める。
- **(b) Grafana 組み込み**: 可視化(Grafana)と説明生成(Bedrock)の責務が混ざり、構成図の読みやすさを損なう。プラグイン開発の学習コストも 60 時間予算に対して過大。
