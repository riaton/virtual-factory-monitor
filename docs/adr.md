## ADR-001: 重複排除は Timestream の冪等書き込みで実装する(追加コンポーネントなし)

**日付**: 2026-07-05 / **状態**: 承認(一度 (a) を仮決定後、同日 (c) に変更)

### 文脈
一点豪華主義として深掘りする非機能は「ネットワーク断・センサー欠損時の挙動」(デバイス側バッファリング → IoT Core 再送 → 重複排除)。このうち重複排除の実装位置が論点だった。

### 選択肢
- (a) IoT Rule → **Lambda** で排除してから Timestream へ書き込む
- (b) 重複ごと Timestream に格納し、クエリ側で除外する
- (c) **Timestream の冪等書き込み**(同一ディメンション+タイムスタンプへの再送を書き込み側で吸収)のみで成立させる

### 決定
**(c) を採用。** 重複耐性は追加コンポーネントではなくデータモデル設計に織り込む——どのディメンションとタイムスタンプの組をべき等キーとするかの設計判断をデータモデル図のコメントとして残す。

### 根拠
1. **60 時間予算との整合**: (a) は Lambda 本体・IAM・Terraform 定義・デバッグの工数が Phase 2 の一点豪華主義枠(5h)を圧迫する。(c) は追加コンポーネントゼロで、浮いた工数をシミュレータの故障モード注入(重複送信・時刻ズレ)の作り込みに回せる。
2. **証明の形**: 「重複を投げても二重計上されない」ことをデモとして実演できれば、証明として (a) と等価である。展示価値は Lambda のコードではなく、(i) データモデル設計の判断コメント、(ii) 異常系シーケンス図、(iii) 故障モード注入による実演、の 3 点で担保する。
3. **アーキテクチャの純度**: 「マネージドサービスの性質を理解し、不要な自前実装を置かない」こと自体がアーキテクトの判断の展示になる。

### 採らなかった案と理由
- **(a) Lambda**: 一度仮決定した(排除ロジックを「読める・テストできるコード」として展示できる価値を評価)。しかし素朴な Lambda 重複排除はステートレスな Lambda 単体では成立せず、既知メッセージ ID の保持に DynamoDB 等の状態ストアが必要になるか、結局 (c) の冪等書き込みに依存するハイブリッドになる。「Lambda を置いたのに排除の実体は Timestream 側」という構成は展示としてかえって説明しづらく、工数増だけが残ると判断し覆した。本番規模での前段処理(Lambda / Kinesis + DynamoDB)の検討は docs/ の非機能設計文書に記載する。
- **(b) クエリ側で除外**: 汚れたデータをそのまま格納し、除外の責務を全クエリ(Grafana・CLI)に分散させる形になる。読者・利用者に負債を渡す設計であり不採用。

### 受け入れたコスト
排除の判断過程がランタイムのコードとして見えない。この弱点はデータモデル図・シーケンス図・実演デモの 3 点で補う(上記根拠 2)。

## ADR-002: Grafana はローカル(docker compose)でセルフホストする

**日付**: 2026-07-05 / **状態**: 承認

### 文脈
可視化層の稼働場所。Amazon Managed Grafana(AMG)とローカルセルフホストが選択肢。

### 選択肢
- (a) Amazon Managed Grafana
- (b) ローカルの docker compose 内でセルフホスト

### 決定
**(b) を採用。** ローカルの Grafana から AWS 上の Timestream / SiteWise へデータソースプラグインでクエリする。

### 根拠
1. **コスト実測の見栄え**: AMG はユーザー単位の月額課金であり、本リポジトリの売りである「コスト明細の実測公開」において固定費が支配的になり、IoT パイプライン本体のコスト感が読者に伝わらなくなる。
2. **再現性**: `terraform apply` + `docker compose up` で全体が立ち上がる、が本リポジトリの再現手順の骨格。可視化までローカルに含めることで読者の追加セットアップをゼロにする。
3. **境界の明示**: 「AWS = Terraform 管理 / ローカル = docker compose」という構成図の境界線が崩れない。

### 採らなかった案と理由
- **(a) AMG**: 本番であれば認証(SSO)・可用性・パッチ運用の観点で AMG が正しい。この判断は「個人検証環境と本番の要件差」の実例として docs/ の非機能設計文書に記載する。

## ADR-003: Bedrock チャットは Python CLI 直結とする(CLI → Timestream クエリ → Bedrock)

**日付**: 2026-07-05 / **状態**: 承認

### 文脈
AI 層(設備の異常を日本語で説明するチャット)のインターフェースとデータ経路。

### 選択肢
- (a) Web UI(Streamlit 等)を立てる
- (b) Grafana のパネル / プラグインに組み込む
- (c) Python CLI が Timestream に直接クエリし、結果を Bedrock に渡して日本語説明を生成する

### 決定
**(c) を採用。**

### 根拠
1. **証明対象の純度**: 本リポジトリの縦貫で証明したいのは「センサーデータ → AI による現場語の説明」というデータ経路であり、UI ではない。スコープの判定基準「この機能がないと読者が誤解するか」に対し、Web UI は No。
2. **工数**: UI 実装・ホスティングの工数(見積もりで 5h 超)を Phase 3 の 10h 枠から排除し、プロンプト設計の試行(4h)に予算を残す。
3. **デモ性**: CLI はターミナル録画で README のデモ GIF にしやすく、90 秒読者への訴求手段としてむしろ有利。

### 採らなかった案と理由
- **(a) Web UI**: 見栄えは上がるが証明対象に寄与しない。README に「Streamlit 等での UI 化は容易」と一行残し、拡張可能性の示唆に留める。
- **(b) Grafana 組み込み**: 可視化(Grafana)と説明生成(Bedrock)の責務が混ざり、構成図の読みやすさを損なう。プラグイン開発の学習コストも 60 時間予算に対して過大。
